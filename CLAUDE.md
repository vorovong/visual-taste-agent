# Visual Taste Agent — 비디자이너의 시각적 취향 발견·체계화 시스템

---

## A. 프로젝트 개요

비디자이너가 자신의 시각적 취향을 발견하고, 디자인 언어로 체계화하여, 모든 시각 매체에서 "나다운 디자인"을 일관되게 만들 수 있게 하는 시스템.

- **최종 사용자**: 김민호 (40대, 개발자, 디자인 비전공, Claude Code + 아이폰 + 텔레그램 사용)
- **개발자**: Claude Code (전권 위임 — end-to-end 자율 구현)
- **현재 문제**: 좋고 싫음은 아는데 "왜 좋은지" 디자인 언어로 표현 못함. AI 도구(V0, Bolt 등)의 결과가 제네릭함 — "내 UI"가 아님. 레퍼런스가 산재하고 체계 없음.
- **해결**: 텔레그램으로 좋아 보이는 것을 던지면 → 자동 분류 → 패턴 추출 → 토큰화 → 디자인 시스템 → 비평까지. 에이전트가 디렉터(나의 취향) × 프로듀서(청중 적합성)를 동시 수행.

---

## B. 사용자 맥락

- 개발자이지만, 프론트엔드 **디자인** 영역은 비전문가.
- AI 증강의 승수효과를 이해함: 기존능력(2) × AI배율(3) = 6. 디자이너(8) × 3 = 24. 핵심 병목은 **평가 능력**.
- 구조/논리에 대한 취향은 뚜렷하지만, **디자인 취향은 별개** — 아직 발견·축적 단계.
- 모바일 사용성이 없으면 시스템 사용 중단. 텔레그램봇이 핵심 입력 채널.
- Opus를 튜터이자 협업자로 활용. 3개 프론티어 LLM 동시 사용.

## C. 행동 오버라이드

이 프로젝트는 **전권 위임** 모드로 진행한다.

- **end-to-end 자율 구현**: 아키텍처, 기술 스택, 구현 순서 모두 AI가 판단하고 실행.
- **판단 근거 기록**: 자율 실행하되, 모든 기술적 판단과 근거를 progress.md에 기록.
- **기존 자산 활용**: 지식 저장소(`~/Library/Mobile Documents/iCloud~md~obsidian/Documents/knowledge/`)의 논의 기록, `~/projects/` 하위의 기존 프로젝트 패턴을 참조.
- **부족한 것은 자체 해결**: 필요한 도구/라이브러리/스크립트가 없으면 직접 개발.
- **사용자 확인이 필요한 것**: 외부 서비스 API 키 입력, 배포, 비용 발생하는 결정만.

## D. 아키텍처

```
수집 (입력)                   처리 (에이전트)              저장 (데이터)
──────────                   ──────────────              ───────────
텔레그램봇  ──→ URL/캡쳐/파일 ──→ [분류 + 캡쳐]  ──→  design-system/references/
  (모바일)       수신              Puppeteer              {id}.md + screenshots/
                                    │
                                    ↓
                             [패턴 추출 엔진]    ──→  design-system/patterns/
                             (Lv.1: 5개+ 시)          {pattern-name}.md
                                    │
                                    ↓
                             [토큰화 제안]       ──→  design-system/tokens/
                             (Lv.2: 패턴 안정)        core.md + contexts/
                                    │
                                    ↓
                             [시스템 문서화]     ──→  design-system/system/
                             (Lv.3: 토큰 확정)        {component}.md
                                    │
                                    ↓
                             [비평 모드]         ──→  사용자에게 피드백
```

**3개 컴포넌트**:
1. **텔레그램봇** — 모바일 수집 채널. URL/캡쳐/파일 수신 → 판정(좋다/싫다) → 맥락 태깅 → 저장 트리거
2. **에이전트 코어** — `.claude/agents/design-system.md`. 분류·패턴추출·토큰화·비평 로직
3. **데이터 저장소** — `design-system/` 폴더. profile.md가 핵심

## I. 세션 프로토콜

새 세션을 시작할 때 반드시 아래 순서를 따른다:

**시작:**
1. **`progress.md`를 읽는다** — 현재 상태, 결정 로그, 최근 세션
2. 필요 시 `planning.md`, `spec.md` 참조
3. 현재 상태를 요약하고, 다음 단계를 실행한다 (전권 위임 모드 — 확인 없이 진행)

**종료:**
1. **progress.md의 "현재 상태" 블록을 최신으로 갱신**한다
2. **해당 세션 기록을 추가**한다 (목표, 완료, 결정 & 근거, 변경 파일, 미해결)
3. 중요한 결정이 있었으면 **핵심 결정 로그에 추가**한다

## K. 핵심 개념 요약 (새 세션용 컨텍스트)

### 에이전트의 존재 이유
비디자이너의 핵심 병목은 **평가 능력**. "별론데 왜인지 모름" 상태를 "이건 정보 밀도가 높아서 별로" 수준으로 끌어올리는 것. 생성 AI보다 **비평 AI**가 먼저 필요하고, 비평보다 **취향 추출**이 선행.

### 취향의 3층 모델
- **1층**: 불변 취향 (색, 여백, 서체, 정보 밀도 — 매체 무관)
- **2층**: 맥락별 취향 (매체: 웹앱/PPT/문서/노션, 용도: 대시보드/랜딩/발표/보고서, 청중: 내부팀/투자자/고객)
- **3층**: 프로젝트별 결정 (그때그때)

### 자동 레벨업 (핵심 메커니즘)
```
Lv.0  빈 상태        → "레퍼런스를 보여주세요"
Lv.1  레퍼런스 5개+  → 패턴 추출 ("이런 경향이 보입니다" → 반박 루프)
Lv.2  패턴 안정화    → 토큰 제안 ("이 팔레트는 어떨까요?")
Lv.3  토큰 확정      → 시스템 문서화 + 코드 출력
Lv.4  시스템 존재    → 비평 모드 + 청중 적합성 판단
```

### 디렉터 × 프로듀서 = 에이전트
대화의 출발점(웹툰 프로듀서/디렉터)이 결론과 연결:
- **디렉터 역할**: 사용자의 취향 프로필 기반 일관성 유지
- **프로듀서 역할**: "이 취향을 이 청중에게 통하게 하려면 어디를 조율해야 하는지" 제안
